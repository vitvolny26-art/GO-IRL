

# GO IRL Bible

# Book VI

## 07 — UX & Interaction Guidelines

Version 1.0

---

# UX Philosophy

GO IRL does not compete for attention.

GO IRL competes for action.

Every interaction must reduce friction between the user and a real-life activity.

Users should spend as little time as possible inside the application.

The application exists only to enable real-life experiences.

---

# UX Mission

The ideal user journey is:

```text
Open App

↓

Find Activity

↓

Join

↓

Close Phone

↓

Meet People
```

Everything that does not support this flow should be questioned.

---

# Design Principles

## 1. One Primary Action

Every screen must have exactly one obvious next action.

Examples:

Home → Join Activity

Activity → Join

Profile → Edit Profile

Settings → Save

---

## 2. No Dead Ends

Every screen must always offer a next meaningful action.

If no activities exist:

Create one.

If waiting list is empty:

Browse more activities.

---

## 3. Large Touch Targets

Minimum touch area:

48×48 px

Buttons should always be comfortable to tap with one hand.

---

## 4. Thumb First

Most users hold phones with one hand.

Primary actions belong in the lower half of the screen whenever possible.

---

## 5. Reduce Thinking

The interface should answer questions before users ask them.

Instead of:

> Activity

Use:

🏐 Volleyball

Instead of:

> Participants

Use:

👥 5 / 8 Players

---

# Motion Principles

Animations should communicate.

Never decorate.

Animations indicate:

Loading

Success

Failure

State change

Navigation

Nothing else.

---

# Animation Duration

Fast:

150 ms

Normal:

250 ms

Complex:

350 ms

Anything longer feels slow.

---

# Navigation

Transitions must feel natural.

Forward:

Slide Left

Back:

Slide Right

Bottom Sheet:

Slide Up

Dismiss:

Slide Down

---

# Loading States

Never show blank screens.

Always use skeleton loaders.

Example:

Activity Card Skeleton

Avatar Skeleton

List Skeleton

Map Skeleton

---

# Empty States

Every empty screen must motivate action.

Example:

No volleyball today.

↓

Be the first to organize a game.

[ Create Activity ]

---

# Error States

Never blame the user.

Bad:

Error 403

Good:

Something went wrong.

Please try again.

---

# Success States

Every important action deserves positive feedback.

Example:

🎉 You're in!

Next:

🗓 Add to Calendar

📍 Open Route

💬 Open Chat

---

# Notifications

Notifications must be useful.

Never send notifications simply to increase engagement.

Allowed:

Reminder

Waiting List

Activity Changed

Friend Joined

Verification

Achievement

Not Allowed:

"We miss you"

"Come back"

"You haven't opened the app"

GO IRL does not manipulate users.

---

# Progressive Disclosure

Show only what users need now.

Hide complexity until required.

Example:

Create Activity

Step 1:

Choose Module

Nothing else.

---

# Colors

Colors communicate meaning.

Green

Success

Blue

Information

Yellow

Warning

Red

Problem

Gray

Inactive

Modules may define accent colors.

---

# Typography

Readable.

Minimal.

No decorative fonts.

Text hierarchy must be obvious.

---

# Cards

Cards are the most important UI element.

Every card answers:

What?

When?

Where?

Who?

Can I join?

Users should decide within 3 seconds.

---

# Maps

Maps support activities.

Maps never replace activity lists.

List First.

Map Second.

---

# Chat

Chat exists only to coordinate.

It is not a messenger.

No stickers.

No channels.

No stories.

No reactions (MVP).

Simple.

Fast.

Temporary.

---

# Calendar

Joining an activity should naturally lead to adding it to the user's calendar.

The application becomes part of the user's real schedule.

---

# Accessibility

Every interaction must support:

Large fonts

High contrast

Screen readers

Keyboard navigation (Web)

Color blindness

---

# Delight Moments

Small moments of joy.

Example:

🎉 First Activity

🏆 New Achievement

🔥 10 Activities Completed

🌍 New City Explored

Celebrate real life.

Not screen activity.

---

# AI Assistance

AI should reduce effort.

Examples:

Suggest title

Improve description

Recommend location

Recommend time

Never replace human decisions.

---

# Emotional Tone

GO IRL speaks like a friendly organizer.

Never corporate.

Never robotic.

Never sarcastic.

Simple.

Positive.

Human.

---

# Microcopy

Buttons

Join

Leave

Create

Invite

Share

Avoid technical language.

---

# Speed Rules

Users should never wait if information can be shown instantly.

Optimistic UI is encouraged.

Rollback only if necessary.

---

# UX KPI

Time to Join

Target:

Less than 15 seconds.

Time to Create

Target:

Less than 30 seconds.

Time to Discover

Target:

Less than 60 seconds.

---

# Ultimate UX Goal

The application should disappear from the user's mind the moment they leave home.

Real life is the destination.

The interface is only the bridge.

---

# UX Commandments

1. Don't make users think.
2. Don't waste users' time.
3. Don't interrupt real life.
4. Don't optimize for screen time.
5. Always encourage action.
6. Reward real experiences.
7. Respect privacy.
8. Celebrate participation.
9. Keep everything simple.
10. Make leaving the app feel like success.

---

# Final UX Principle

> The best interaction with GO IRL is the one that ends with people meeting in person.
