BUG-1: Неверное имя правила ESLint (explicit-function-return-types)
------------------------------------------------------------------------
ДО (.eslintrc.json):
  "@typescript-eslint/explicit-function-return-types": "warn"

ПОСЛЕ:
  "@typescript-eslint/explicit-function-return-type": "warn"

Почему старый код неправильный:
  Правила "explicit-function-return-types" (с "s") не существует в
  @typescript-eslint/eslint-plugin. ESLint тихо игнорировал неизвестное
  правило — функции без явного возвращаемого типа не проверялись.

Почему новый код правильный:
  "explicit-function-return-type" — реальное правило пакета. Теперь
  правило активно.

Может ли сломать:
  Нет. Только усиливает lint-проверку. Могут появиться новые предупреждения
  на уже написанном коде — это ожидаемо.

Тесты:
  pnpm run lint в /backend на функции без явного типа возврата должна
  выдавать warn.


BUG-2: CI — неверная команда проверки форматирования
------------------------------------------------------------------------
ДО (.github/workflows/ci.yml):
  - name: Check formatting
    run: pnpm run format --check

ПОСЛЕ:
  - name: Check formatting
    run: pnpm run format:check

  (добавлен скрипт в package.json:)
  "format:check": "prettier --check \"**/*.{ts,tsx,js,jsx,json,md,css}\" --ignore-path .gitignore"

Почему старый код неправильный:
  Скрипт "format" запускает "turbo run format", который в каждом пакете
  вызывает "prettier --write". Флаг --check передавался в turbo, а не в
  prettier. Turbo его игнорировал — шаг проходил всегда, даже с неправильно
  отформатированным кодом.

Почему новый код правильный:
  Скрипт format:check вызывает prettier --check напрямую и завершается с
  ненулевым exit code при нарушениях форматирования.

Может ли сломать:
  Нет. CI-only.

Тесты:
  Закоммитить файл с неправильным отступом → CI шаг "Check formatting"
  должен упасть с exit code 1.

Почему новый код правильный:

BUG-5: Неверный путь к скомпилированному файлу dist/src/main.js
------------------------------------------------------------------------
ДО (Dockerfile.backend):
  CMD ["node", "dist/src/main.js"]

ДО (backend/package.json):
  "start": "node dist/src/main.js"

ПОСЛЕ:
  CMD ["node", "dist/main.js"]
  "start": "node dist/main.js"

Почему старый код неправильный:
  В backend/tsconfig.json задано:
    "rootDir": "./src"
    "outDir": "./dist"
  TypeScript компилирует src/main.ts → dist/main.js.
  Путь dist/src/main.js никогда не создаётся. Контейнер падал бы с
  "Error: Cannot find module".

Почему новый код правильный:
  dist/main.js — реальный путь к скомпилированному файлу.

Может ли сломать:
  Нет. Старый путь был гарантированно нерабочим.

Тесты:
  cd backend && pnpm run build && node dist/main.js — должен запустить сервер.


BUG-6: pnpm-lock.yaml в .gitignore
------------------------------------------------------------------------
ДО (.gitignore, строки 39-40):

  pnpm-lock.yaml

ПОСЛЕ:

Может ли сломать:

Тесты:
  CI шаг pnpm install --frozen-lockfile проходит без ошибок.


BUG-7: telegramId: bigint во фронтенд-типах
------------------------------------------------------------------------
ДО (apps/telegram-miniapp/src/types/index.ts):
  export interface User {
    id: string;
    telegramId: bigint;
    ...
  }

ДО (apps/telegram-miniapp/src/services/api.ts):
  interface AuthResponse {
    data?: {
      user: {
        telegramId: bigint;
        ...
      };
    };
  }

ПОСЛЕ:
  telegramId: string;

Почему старый код неправильный:
  JSON не поддерживает BigInt. JSON.parse() не возвращает bigint — он
  возвращает number. Для больших Telegram ID (> 2^53) это вызывает потерю
  точности. Бэкенд явно сериализует telegramId как строку через
  user.telegramId.toString() в serializeUser()
  (backend/src/adapters/http/routes/auth.ts). Тип bigint во фронтенде
  был принципиально несовместим с реальным ответом API.

Почему новый код правильный:
  string точно соответствует тому, что возвращает API.

Может ли сломать:
  Нет. Тип стал точнее.

Тесты:
  authenticateWithTelegram() → user.telegramId должна быть строка.


BUG-8: useActivityDetails — отсутствующий useEffect
------------------------------------------------------------------------
ДО (apps/telegram-miniapp/src/hooks/useActivityDetails.ts):
  import { useState } from 'react';

  export function useActivityDetails(activityId: string) {
    const [activity, setActivity] = useState<Activity | null>(null);
    const [loading, setLoading] = useState(true);  // true, но фетч никогда не стартует
    const [error, setError] = useState<string | null>(null);

    const fetchActivity = async () => { ... };

    return { activity, loading, error, refetch: fetchActivity };
    // useEffect отсутствует — fetchActivity никогда не вызывается автоматически
  }

ПОСЛЕ:
  import { useState, useEffect } from 'react';

  export function useActivityDetails(activityId: string) {
    const [loading, setLoading] = useState(false);  // false, пока фетч не начнётся

    useEffect(() => {
      fetchActivity();
      // eslint-disable-next-line react-hooks/exhaustive-deps
    }, [activityId]);

    return { activity, loading, error, refetch: fetchActivity };
  }

Почему старый код неправильный:
  Хук возвращал loading: true и activity: null навсегда. ActivityDetailsPage
  показывал бы бесконечный лоадер — данные никогда не загружались.

Почему новый код правильный:
  useEffect автоматически вызывает fetchActivity() при монтировании и при
  изменении activityId.

Может ли сломать:
  Нет. Исправляет сломанное поведение.

Тесты:
  Render ActivityDetailsPage с activityId → должен отрисовать данные
  активности (не застрять на загрузке).

BUG-10: CI — отсутствие явного permissions блока
------------------------------------------------------------------------
ДО (.github/workflows/ci.yml):
  (нет блока permissions)

ПОСЛЕ:
  permissions:
    contents: read   # глобально

  jobs:
    lint:
      permissions:
        contents: read
    typecheck:
      permissions:
        contents: read
    build:
      permissions:
        contents: read
    docker-compose-validate:
      permissions:
        contents: read

Почему старый код неправильный:
  Без явного блока permissions GitHub Actions наследует дефолтные настройки
  репозитория. При настройке read-write по умолчанию скомпрометированная
  зависимость (supply chain attack) могла бы записывать в репозиторий.

Почему новый код правильный:
  Принцип наименьших привилегий. CI требует только contents: read.

Может ли сломать:
  Нет. Джобы только читают код и строят его.

Тесты:
  CI проходит успешно.
